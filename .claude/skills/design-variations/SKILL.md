  side-by-side comparison. Use whenever the user wants design options, mockups,
  or "a few versions" of a component, section, or page — e.g. "תעשי לי כמה
  וריאציות ל...", "וריאציות עיצוב", "3 גרסאות לכרטיס", "design options for a

  it first peeks at the current repo (Tailwind config, global CSS, existing
  components, CLAUDE.md conventions) to match the project's real design language
  — colors, fonts, radius, text direction (RTL/LTR), language, and any cultural
  or brand rules — so the chosen variation ports straight into real code. If no
  project style is found, it falls back to clean modern defaults and says so.

# מחולל וריאציות עיצוב (Design Variations)

המטרה: המשתמש מבקש לעצב אלמנט כלשהו (כרטיס, כותרת Hero, טופס, באנר, עמוד…),
ואתה מפיק **כמה וריאציות שונות מהותית זו מזו** ומרנדר אותן ל**דף HTML אחד עצמאי**
שנפתח בדפדפן להשוואה מהירה. הכלי גנרי לכל פרויקט, אבל **לומד את שפת העיצוב של
הפרויקט הנוכחי** לפני שהוא מייצר, כדי שמה שנבחר יתתרגם ישר לקוד אמיתי.

ברירת מחדל: **4 וריאציות**. אם המשתמש נקב במספר — כבד אותו.

---

## שלב 1 — הבן את הבקשה (שאלה אחת לכל היותר)

חלץ מהבקשה: מה האלמנט? הקשר/עמוד? אילוצים מפורשים (תוכן, מותג, מצב)? כמה וריאציות?

אם משהו קריטי חסר — שאל **שאלה אחת קצרה** ולא יותר. אם הבקשה ברורה מספיק, אל תשאל —
תייצר. עדיף להראות 4 כיוונים ולתת למשתמש לכוון מהם, מאשר לתשאל.

## שלב 2 — למד את שפת העיצוב של הפרויקט (אוטומטי, לפני יצירה)

**זה מה שהופך את התוצאה לרלוונטית ולא גנרית.** סרוק במהירות (Read/Grep/Glob, לא לעומק):

1. **טוקנים:** `tailwind.config.{js,ts,cjs,mjs}` — צבעים מותאמים, `fontFamily`,
   `borderRadius`, spacing, breakpoints. אם אין Tailwind — חפש CSS variables/theme
   בקובץ globals/tokens.
2. **סגנון גלובלי:** `globals.css` / `app.css` / `index.css` — פונט, כללי כותרות,
   רקע ברירת מחדל, כללי צבע גורפים (למשל כפייה של צבע טקסט לכותרות).
3. **אידיומים קיימים:** 1–2 קומפוננטות UI קיימות (Button, Card, Input…) — פינות,
   צללים, מרווחים, סגנון CTA — כדי לחקות ולא להמציא.
4. **כיוון ושפה:** `dir="rtl"`/`lang` ב-layout, ושפת ה-copy הקיים.
5. **כללי מותג/תרבות:** `CLAUDE.md` / README / קובצי spec — אילוצים שחובה לכבד
   (למשל RTL מלא, לשון זכר בלבד, איסור תמונות פנים/אנשים, סימון "מודעה", טון מכובד).
   **כל אילוץ כזה גובר — כל וריאציה חייבת לציית לו.**

צור לעצמך "כרטיס מותג" קצר בראש (פלטה, פונט, radius, direction, language, rules).
אם הריפו אינו web / אין שום סגנון — השתמש בברירות מחדל מודרניות ונקיות, וציין
במפורש למשתמש שלא נמצאה שפת עיצוב פרויקטלית והשתמשת בברירת מחדל.

## שלב 3 — בחר צירי שונוּת (כל וריאציה על ציר אחר)

**חוק ברזל:** אין שתי וריאציות שהן "אותו דבר עם שינוי קטן". כל וריאציה חייבת
להיבדל ב-**layout ראשי ו/או קונספט** — לא רק בצבע כפתור. תן לכל וריאציה שם-קונספט
קצר (למשל "מינימלי-אוורירי", "כרטיס דחוס עם דגש CTA", "היררכיה טיפוגרפית נועזת").

בחר לכל וריאציה ציר דומיננטי שונה מתוך (או משלב של) התפריט הזה:

- **מבנה/Layout** — עמודה יחידה / split / רשת / מרכוז / hero-side.
- **היררכיה ויזואלית** — מה שולט: כותרת ענקית? תמונה/צורה? ה-CTA?
- **דגש צבע** — ניטרלי עם אקסנט בודד / מונוכרום עשיר / גבולות בלבד ללא מילוי.
- **צפיפות** — אוורירי מרווח מול קומפקטי-אינפורמטיבי.
- **קישוט** — שטוח נקי / צללים ועומק / קווי מסגרת / מרקע עדין.
- **סגנון CTA** — כפתור מלא בולט / משני עדין / קישור טקסטואלי / כפול.
- **טיפוגרפיה** — סקאלה שמרנית מול קונטרסט גדלים נועז.

ודא שהסט מכסה כיוונים באמת שונים (למשל: 1 אוורירי-מינימלי, 1 דחוס-תכליתי,
1 עם דגש ויזואלי/צורה, 1 עם היררכיה טיפוגרפית נועזת) — לא ארבע וריאציות על אותה תבנית.

## שלב 4 — בנה את גלריית ה-HTML

השתמש בשלד המוכן: [assets/gallery-template.html](assets/gallery-template.html).
קרא אותו, מלא את ה-placeholders, והזרק את גוף כל וריאציה. עקרונות:

- **עצמאי לגמרי** — Tailwind Play CDN (`https://cdn.tailwindcss.com`) + הפונט של
  הפרויקט מ-Google Fonts. בלי build, נפתח מיד בדאבל-קליק.
- **כבד את הטוקנים שזוהו** — הזרק את הפלטה ל-`tailwind.config` ה-inline שבתבנית,
  והגדר `dir`/`lang` לפי הפרויקט. ה-canvas חייב להרגיש כמו האתר האמיתי.
- **כל וריאציה במסגרת מסומנת** — badge "וריאציה N", שם-הקונספט, הציר שהיא בודקת,
  ומשפט רציונל קצר (אחד). מתחת — ה-canvas עם העיצוב עצמו.
- **מתג רוחב דסקטופ/מובייל** כבר בתבנית (JS פשוט) — ודא שהוא עובד על כל ה-canvases.
- **תוכן דמו אמין** — טקסט עברי/אנגלי אמיתי לפי שפת הפרויקט ולשון תקינה לפי כלליו
  (למשל לשון זכר יחיד אם זה כלל הפרויקט). ללא lorem ipsum. **ללא תמונות פנים/אנשים**
  אם זה כלל הפרויקט — השתמש באייקונים inline (SVG/Phosphor/Heroicons), צורות, או
  placeholders גיאומטריים.
- **שמירה:** צור תיקיית `design-lab/` בשורש הריפו אם אינה קיימת, ושמור בשם
  `design-lab/<slug>--<timestamp>.html` (slug באנגלית מהבקשה, timestamp
  `yyyyMMdd-HHmm` מהשעון — קבל אותו דרך shell כדי שלא יידרס וגם שומר היסטוריה).

## שלב 5 — פתח והצג

- אם כלי `browser_*` (Playwright MCP) זמינים — נווט ל-`file:///…`, קח `browser_snapshot`
  לאימות שהכול רונדר, וצלם `browser_take_screenshot` לתצוגה מהירה בצ'אט.
- אם לא — פתח בדפדפן המערכת: Windows `Start-Process <path>` / macOS `open` /
  Linux `xdg-open`.
- בצ'אט: תן את הנתיב הקליקבילי + רשימה ממוספרת של הוריאציות (שם-קונספט + משפט לכל אחת).

## שלב 6 — בחירה → מימוש

בקש מהמשתמש לענות במספר. אחרי הבחירה, הצע/בצע מימוש של הוריאציה כקומפוננטה אמיתית
בפרויקט (React/Vue/וכו') תוך שימוש בקומפוננטות ובטוקנים הקיימים — לא העתקה של ה-CDN HTML.

## עקרונות מנחים

- **שונוּת אמיתית לפני ליטוש** — עדיף 4 כיוונים נבדלים בבירור מ-4 גרסאות כמעט-זהות.
- **נאמנות למותג** — משתמש בטוקנים ובכללים של הפרויקט; לא ממציא צבעים/פונטים משלו.
- **אילוצי הפרויקט הם קודקוד** — RTL, שפה, לשון, איסורי דימויים, טון — כל וריאציה מצייתת.
- **מהיר להשוואה** — דף אחד, עצמאי, נפתח מיד, מסמן בבירור מה כל וריאציה מנסה.
- **לא מלכלך את הריפו** — הכול ב-`design-lab/` (gitignored). לא נוגע בקוד האפליקציה
  עד שהמשתמש בוחר וריאציה.
