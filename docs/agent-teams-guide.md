# מדריך צוותי סוכנים — דער שטעלע

איך להשתמש הכי טוב בצוות הסוכנים של הפרויקט. מבוסס על התיעוד הרשמי:
<https://code.claude.com/docs/en/agent-teams>

---

## מה זה בכלל

**צוותי סוכנים (Agent Teams)** = כמה מופעים של Claude Code שעובדים יחד. סשן אחד הוא
**המוביל** (Lead) — הוא מקים חברי צוות (teammates), מחלק משימות דרך **רשימת משימות
משותפת**, ומסנתז את הממצאים. כל teammate הוא סשן עצמאי עם חלון קונטקסט משלו, ויכול
לתקשר ישירות עם חברי צוות אחרים.

**זה שונה מ-subagents:** subagent רק מדווח בחזרה למוביל ולא מדבר עם אחרים. ב-Agent
Teams חברי הצוות חולקים רשימת משימות, תופסים עבודה, ומדברים זה עם זה — ואת יכולה
לפתוח לכל אחד מהם תמליל ולכתוב לו ישירות.

> ⚠️ **ניסיוני.** הפיצ'ר מסומן experimental. יש מגבלות ידועות (ראי בסוף). צורך
> **משמעותית יותר טוקנים** מסשן בודד — כל teammate הוא Claude נפרד.

---

## איך זה מוגדר אצלך

1. **הדגל כבר הופעל** — הוספנו ל-`.claude/settings.json` שבפרויקט:
   ```json
   { "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }
   ```
2. **צריך restart ל-Claude Code** כדי שהדגל ייטען (משתני `env` נקראים בעליית הסשן).
3. ⚠️ **Windows + טרמינל VSCode = מצב `in-process` בלבד.** מצב "חלונות מפוצלים"
   (split panes) דורש tmux או iTerm2, ולא נתמך בטרמינל של VSCode / Windows Terminal /
   Ghostty. אצלך כל חברי הצוות ירוצו בתוך הטרמינל הראשי, ומופיעים ב**חלונית הסוכנים**
   מתחת לשורת הקלט.

**חשוב לדעת:** 7 הסוכנים ב-`.claude/agents/` עובדים **גם בלי restart** — אני יכולה
להפעיל אותם כבר עכשיו כ-subagents רגילים (כלי Agent), אפילו במקביל. ה-restart נחוץ רק
לחוויית ה-Teams המלאה (חלונית, מסירת הודעות בין חברי צוות, רשימת משימות משותפת).

## הרוסטר — 7 חברי הצוות

הגדרות הסוכנים יושבות ב-`.claude/agents/*.md`. כל אחד מכבד את רשימת ה-`tools` וה-`model`
שלו, וגוף ההגדרה מתווסף למערכת-הפרומפט שלו. כולם טוענים אוטומטית את `CLAUDE.md`,
הסקילים וה-MCP של הפרויקט.

| סוכן                  | תפקיד                                    | מתי להפעיל                   | הרשאות            |
| --------------------- | ---------------------------------------- | ---------------------------- | ----------------- |
| **feature-developer** | בונה/מתקן קוד לפי האיפיון והקונבנציות    | פיתוח פיצ'ר, תיקון באג       | כתיבה מלאה        |
| **qa-tester**         | מריץ Vitest/lint/build, בודק end-to-end  | אחרי כל שינוי קוד            | קריאה + Bash      |
| **security-reviewer** | JWT, escaping, פרטיות, אחסון קו"ח        | לפני מיזוג                   | קריאה + סקילים    |
| **design-reviewer**   | RTL, Tailwind, נגישות, צניעות (בלי פנים) | שינוי UI                     | קריאה + צילום מסך |
| **cultural-guardian** | לשון זכר, בלי פנים, שבת, מודל התיווך     | טקסט/תמונות/מיילים/זרימת קשר | קריאה בלבד        |
| **spec-guardian**     | עמלות, זרימות משרה/מעסיק, פרטיות         | לוגיקה עסקית                 | קריאה בלבד        |
| **copywriter**        | ניסוח עברי בקול המותג (der-shtele-copy)  | כותרות, מיילים, microcopy    | עריכת טקסט        |

הבודקים הם **קוראי-בלבד** בכוונה — הם מדווחים, ו-`feature-developer` מתקן. כך אין
התנגשות קבצים בין חברי צוות.

## איך מפעילים אותם

מתארים למוביל בשפה טבעית מה רוצים, **ומזכירים את שם הסוכן**. דוגמאות מוכנות להדבקה:

**סקירה מקבילה של פיצ'ר קיים (הכי מומלץ להתחלה — בלי כתיבת קוד):**

```
הקם צוות שיסקור את השינויים האחרונים ב-branch:
- teammate מסוג cultural-guardian — יבדוק לשון זכר, צניעות, שבת ומודל התיווך
- teammate מסוג spec-guardian — יבדוק תאימות ל-spec-v3.1 וחוקי העסק
- teammate מסוג security-reviewer — יבדוק אבטחה ופרטיות
כל אחד יסקור בנפרד וידווח ממצאים. חכה שכולם יסיימו לפני שאתה מסנתז.
```

**בניית פיצ'ר מקצה לקצה:**

```
בנה את הפיצ'ר X (פרטים: ...). הקם teammate מסוג feature-developer שיבנה אותו.
כשהוא מסיים, הקם במקביל cultural-guardian, spec-guardian ו-design-reviewer שיסקרו,
ו-qa-tester שיריץ בדיקות. אַל תתחיל לסנתז לפני שכולם סיימו.
```

**ניסוח:**

```
הקם teammate מסוג copywriter שיכתוב כותרת + תת-כותרת לדף הנחיתה למעסיקים.
```

טיפ: תני לכל teammate שם קבוע בהוראה ("קרא לו bugcheck") כדי שתוכלי להתייחס אליו אחר כך.

## חלונית הסוכנים — מקשים (מצב in-process)

חברי הצוות מופיעים מתחת לשורת הקלט:

| מקש        | פעולה                                     |
| ---------- | ----------------------------------------- |
| **↑ / ↓**  | בחירת teammate                            |
| **Enter**  | פתיחת התמליל שלו + כתיבת הודעה ישירה אליו |
| **Esc**    | הפסקת התור הנוכחי של ה-teammate שנבחר     |
| **x**      | עצירת ה-teammate שנבחר                    |
| **Ctrl+T** | הצגה/הסתרה של רשימת המשימות               |

שורה של teammate שהתבטל (idle) נעלמת אחרי 30 שניות — הוא עדיין רץ; שלחי לו הודעה בשם
כדי להחזיר אותו.

## שיטות עבודה מומלצות (best practices)

- **3–5 חברי צוט** לרוב המשימות; ~5–6 משימות לכל teammate. שלושה ממוקדים עדיפים על
  חמישה מפוזרים.
- **תני קונטקסט בהוראת ההקמה.** teammate טוען את `CLAUDE.md` והסקילים, אבל **לא**
  יורש את היסטוריית השיחה של המוביל. פרטי משימה → בתוך הפרומפט.
- **התחילי מ-Research/Review**, לא מכתיבת קוד — גבולות ברורים, פחות תיאום.
- **מנעי התנגשות קבצים** — כל teammate על סט קבצים אחר. (הבודקים שלנו קוראי-בלבד, אז
  הסיכון קטן.)
- **חכי שיסיימו.** אם המוביל מתחיל לעבוד בעצמו במקום להאציל: "חכה שחברי הצוות יסיימו".
- **נטרי והכווני** — היכנסי לתמלילים, תקני כיוון שלא עובד, סנתזי תוך כדי.
- **אישור תוכנית (אופציונלי):** למשימות מסוכנות הוסיפי "דרוש אישור תוכנית לפני שינויים"
  — ה-teammate יעבוד ב-plan mode עד שהמוביל יאשר.
- **מודל חברי הצוות:** ב-`/config` → **Default teammate model**. מומלץ
  **Default (leader's model)** כדי שיעבדו על אותו מודל כמוך.

## זרימת עבודה מומלצת לפרויקט הזה

לבניית פיצ'ר טיפוסי:

1. **feature-developer** בונה, קורא קודם את `docs/spec-v3.1.md`.
2. במקביל, ברגע שיש קוד — **ארבעת השומרים סוקרים בו-זמנית**:
   `cultural-guardian` (הכי קריטי כאן), `spec-guardian`, `security-reviewer`,
   `design-reviewer`.
3. **qa-tester** מריץ Vitest + lint + build.
4. המוביל מסנתז, ה-feature-developer מתקן לפי הממצאים.
5. ל-copy — **copywriter** דרך der-shtele-copy.

הערך הכי גדול: `cultural-guardian` + `spec-guardian` תופסים דברים שסקירת קוד רגילה
מפספסת (לשון זכר, פנים בתמונות, גבייה מוקדמת של עמלה, דליפת פרטי קשר).

## אכיפה אוטומטית (אופציונלי, מתקדם)

אפשר לחבר **hooks** ב-`settings.json` שיאכפו שערים:

- `TeammateIdle` — רץ כשה-teammate עומד לסיים; יציאה בקוד 2 מחזירה אותו לעבודה עם משוב.
- `TaskCreated` / `TaskCompleted` — חוסמים יצירה/סגירה של משימה עם משוב (קוד 2).

למשל: hook שמריץ `npm run lint` לפני שמשימה נסגרת. תגידי לי אם את רוצה ואגדיר.

---

## מגבלות שכדאי להכיר

- **אין שחזור teammates ב-in-process:** `/resume` ו-`/rewind` לא משחזרים חברי צוות.
  אחרי resume — בקשי מהמוביל להקים חדשים.
- **סטטוס משימה יכול לפגר** — לפעמים teammate לא מסמן משימה כהושלמה. אם משהו תקוע,
  בדקי אם באמת נגמר ועדכני ידנית / בקשי מהמוביל לדחוף.
- **צוות אחד לסשן**, **בלי צוותים מקוננים** (teammate לא מקים teammates), והמוביל קבוע.
- **כיבוי איטי** — teammate מסיים את הבקשה הנוכחית לפני שנסגר.

---

## סיכום מהיר

1. עשי **restart** ל-Claude Code (טעינת הדגל).
2. ב-`/config` קבעי **Default teammate model → Default (leader's model)**.
3. התחילי מ**סקירה מקבילה** של שינוי קיים עם 3 השומרים.
4. משם — פיצ'ר מלא: dev → 4 סוקרים במקביל → qa → סנתוז.

עד ה-restart — אני יכולה להריץ את אותם סוכנים כ-subagents כבר עכשיו.
