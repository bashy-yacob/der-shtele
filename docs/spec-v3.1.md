## 1. מודל עסקי

## 3. הרשמה ואזור אישי — מועמד

### 3.1 תהליך הרשמה

הרשמה היא תנאי לשליחת קורות חיים. הגולש יראה את תוכן המשרות חופשי, אבל בלחיצה על
"הגש מועמדות" יופנה להרשמה אם אינו מחובר.

| שדה הרשמה                | סוג     | חובה?     | הערות                            |
| ------------------------ | ------- | --------- | -------------------------------- |
| שם מלא                   | טקסט    | כן        |                                  |
| מייל                     | מייל    | כן        | משמש לכניסה ולעדכונים            |
| טלפון                    | מספר    | כן        |                                  |
| סיסמה                    | סיסמה   | כן        | אימות כפול                       |
| אישור קבלת עדכונים במייל | צ'קבוקס | כן — חובה | הסכמה מפורשת לפני הרשמה (Opt-in) |

> ⚡ אישור קבלת מיילים — צ'קבוקס חובה בהרשמה. ללא סימון — לא ניתן להשלים הרשמה.
> זה מאפשר לשלוח עדכונים, ניוזלטר ותוכן שיווקי כדין.

💡 לאחר הרשמה — מייל אישור נשלח מיידית לאימות הכתובת.

### 3.2 פרטי אזור אישי — מועמד

| לשונית       | תכונה         | תיאור                                          |
| ------------ | ------------- | ---------------------------------------------- |
| פרופיל       | עדכון פרטים   | שם, טלפון, מייל, שינוי סיסמה                   |
| פרופיל       | קורות חיים    | הצגת הקובץ הנוכחי + אפשרות החלפה               |
| ההגשות שלי   | מעקב הגשות    | רשימת משרות שהוגשה אליהן + סטטוס כל אחת        |
| משרות שמורות | משרות שסימנתי | רשימת משרות שהמועמד שמר — עם כפתור "הגש עכשיו" |
| הגדרות       | עדכוני מייל   | ניהול העדפות — אפשרות לבטל הסכמה (Opt-out)     |

💡 משרה שנסגרה ונמצאת ברשימת המשרות השמורות — תסומן כ"לא פעילה" ולא תוסר אוטומטית.

## 6. פורטל מעסיקים

מעסיקים מקבלים גישה נפרדת לפורטל ייעודי — לא דרך האתר הציבורי. מעסיק חדש **נרשם
עצמאית** מעמוד הנחיתה למעסיקים (שם חברה, איש קשר, טלפון, מייל, סיסמה + אישור
opt-in חובה). ההרשמה היא **בקשת גישה**: נפתח חשבון בסטטוס `pending` שיכול להתחבר
רק למסך "ממתין לאישור" — לא לפרסם משרות ולא לראות מועמדים. הצוות מקבל התראה, מאמת
(טלפונית לפי שיקול דעתו), ומאשר בדשבורד → סטטוס `approved`, ואז המעסיק רשאי לפרסם.
גולש שאינו מחובר רואה רק את עמוד הנחיתה שמזמין להירשם. הצוות יכול גם להפיק חשבון
ידנית מהדשבורד (נוצר מאושר אוטומטית).

### 6.0 מצבי בקשת גישה (מעסיק)

| סטטוס      | מי יוצר                   | מה מותר                                              |
| ---------- | ------------------------- | ---------------------------------------------------- |
| `pending`  | הרשמה עצמית               | התחברות + מסך המתנה + פנייה לצוות. ללא פרסום/מועמדים |
| `approved` | אישור הצוות / יצירה ידנית | פורטל מלא — פרסום משרות, צפייה במועמדים שהוצגו       |
| `rejected` | דחיית הצוות               | התחברות → מסך "הבקשה לא אושרה" בלבד                  |

### 6.1 טופס פרסום משרה

| שדה                | גלוי לציבור?   | תיאור                          |
| ------------------ | -------------- | ------------------------------ |
| שם החברה           | לא             | נשמר פנימית בלבד               |
| איש קשר + טלפון    | לא             | לשימוש הצוות                   |
| תיאור פנימי מלא    | לא             | דרישות מפורטות, תנאים, הערות   |
| תיאור ציבורי       | כן             | הטקסט האנונימי שעולה לאתר      |
| תחום + אזור + היקף | כן             | לסינון בלוח המשרות             |
| ניסיון נדרש (שנים) | כן             | מוצג בכרטיס ובסינון — חדש      |
| שכר מוצע           | לא (כללי בלבד) | מוצג כ"שכר מתגמל" או טווח כללי |

### 6.2 מה המעסיק רואה בפורטל

| מסך             | תיאור                                          |
| --------------- | ---------------------------------------------- |
| המשרות שלי      | רשימת כל המשרות — פעילות / מושהות / סגורות     |
| מועמדים שהוצגו  | רשימת מועמדים שהוגשו לכל משרה — ע"י הצוות בלבד |
| סטטוס מועמד     | הוצג / בתהליך ראיון / נדחה / גויס              |
| תקשורת עם הצוות | שדה הערות / פנייה                              |
| חשבוניות ועמלות | היסטוריית תשלומים                              |

💡 מעסיק לא רואה פרטי קשר של מועמד — הכל מגיע דרך הצוות.

## 8. מיילים והתראות

### 8.1 הרשמה לעדכונים — Opt-in

כל מועמד רשום מאשר בהרשמה קבלת מיילים. זהו תנאי לרישום, לא אפשרות. ניתן לבטל בכל
עת מהאזור האישי (Opt-out).

> ⚡ חשוב מבחינה משפטית: שמירת תאריך ושעת ההסכמה לכל משתמש — מחויב לפי חוק הספאם הישראלי.

### 8.2 סוגי מיילים

| סוג מייל              | טריגר                      | מי מקבל          | ניתן לביטול?    |
| --------------------- | -------------------------- | ---------------- | --------------- |
| אישור הרשמה           | מיד לאחר יצירת חשבון       | מועמד חדש        | לא — מייל מערכת |
| אימות כתובת מייל      | הרשמה                      | מועמד חדש        | לא — מייל מערכת |
| אישור קבלת מועמדות    | הגשת טופס                  | מועמד            | לא — מייל מערכת |
| עדכון סטטוס           | שינוי סטטוס ידני ע"י צוות  | מועמד            | כן              |
| משרות חדשות רלוונטיות | ידני / שבועי / כשעולה משרה | מועמדים עם הסכמה | כן              |
| ניוזלטר / תוכן        | לפי שיקול הצוות            | מועמדים עם הסכמה | כן              |
| התראה פנימית — צוות   | הגשת טופס                  | צוות             | לא              |
| תזכורת עמלה           | 90 יום מגיוס               | צוות             | לא              |
| תזכורת שיחה           | תזכורת ידנית               | צוות             | לא              |

💡 כל המיילים יישלחו דרך Resend או Nodemailer — ללא שליחה בשבת ויו"ט.

### 8.3 ניהול רשימת התפוצה מהדשבורד

הצוות יכול לראות בדשבורד את מספר המנויים הפעילים, לייצא רשימה, ולשלוח מייל ידני
לכלל המנויים או לקבוצה מסוננת.

| פעולה            | תיאור                                         |
| ---------------- | --------------------------------------------- |
| צפייה במנויים    | רשימת מועמדים עם הסכמת מייל + תאריך הרשמה     |
| שליחה ידנית      | שליחת תוכן / עדכון לכלל המנויים               |
| סינון לפני שליחה | לפי תחום / אזור / סטטוס — שליחה ממוקדת        |
| ייצוא            | ייצוא רשימה ל-CSV לשימוש חיצוני (מחייב הסכמה) |

### 14.1 ארכיטקטורה בפועל — Monorepo

הפרויקט עבר מאפליקציית Next.js בודדת ל-**monorepo עם npm workspaces**:

```
der-shtele/
├── package.json        # workspaces: ["frontend", "backend"]
├── frontend/           # Next.js 14 — אתר ציבורי + אזורים אישיים
└── backend/            # NestJS — לוגיקה עסקית, DB, מיילים, אחסון
```

**עיקרון:** ה-frontend לא ניגש ל-DB ישירות. כל הנתונים עוברים דרך ה-API של ה-backend
(`frontend/src/lib/api.ts` + פרוקסי דק ב-`frontend/src/app/api/*`). זה אוכף את הפרדת
המידע הציבורי/פנימי.

### 14.2 מה בנוי ועובד ✅

| רכיב                                                                                                           | סטטוס                        | מיקום                                   |
| -------------------------------------------------------------------------------------------------------------- | ---------------------------- | --------------------------------------- |
| מבנה monorepo + workspaces                                                                                     | ✅ הותקן, `npm install` עובר | שורש                                    |
| **Frontend** — מתקמפל (`tsc` נקי) + **רץ על :3000**                                                            | ✅                           | `frontend/`                             |
| כל הדפים הציבוריים (בית/jobs/about/contact)                                                                    | ✅ נטענים 200                | `frontend/src/app/`                     |
| עמודי auth (login/register עם **opt-in חובה**)                                                                 | ✅ בנויים                    | `frontend/src/app/(auth)/`              |
| אזור אישי (פרופיל/הגשות/שמורות/הגדרות) + שער כניסה                                                             | ✅ בנוי                      | `frontend/src/app/account/`             |
| רכיבי UI (Button/Input/Card) + hook useAuth                                                                    | ✅                           | `frontend/src/components/ui/`, `hooks/` |
| **Backend** — NestJS, מתקמפל (`tsc` נקי)                                                                       | ✅                           | `backend/`                              |
| 9 מודולים: jobs, candidates, employers, applications, placements, commissions, reminders, contact, auth, email | ✅ בנויים                    | `backend/src/modules/`                  |
| לוגיקה רוחבית: status-machine, shabbat, commission (ערבות 3 חודשים), storage (Supabase), guards (JWT+roles)    | ✅                           | `backend/src/common/`                   |
| Prisma: מודל `User` חדש + שדה opt-in למועמד + seed                                                             | ✅ נכתב                      | `backend/prisma/`                       |

### 14.3 מה עדיין חסר / טרם הורץ 🔲

| משימה                           | פירוט                                                                                                              |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| **הרצת ה-backend**              | טרם הורץ — דורש `DATABASE_URL` (PostgreSQL) אמיתי                                                                  |
| **מיגרציית DB**                 | נדרשת עבור מודל `User` החדש: `npm run prisma:migrate`                                                              |
| **מילוי env**                   | `backend/.env` + `frontend/.env.local` צריכים ערכים אמיתיים (DB, JWT*SECRET, SUPABASE*\_, SMTP\_\_)                |
| **חיבור לוח המשרות ל-backend**  | כרגע `frontend/src/app/jobs/*` עדיין משתמשים ב-`MOCK_JOBS` (3 דמה); `lib/api.ts` כבר מוכן לחיבור אמיתי עם fallback |
| **CRM / דשבורד מנהלים (שלב ג)** | ה-API קיים ב-backend; ממשק הצוות עדיין לא נבנה                                                                     |
| **git commit**                  | כל השינויים ב-working tree, **טרם בוצע commit**                                                                    |

### 14.4 איך מריצים (לשיחה הבאה)

```bash
cp .env.example backend/.env        # מלא DATABASE_URL, JWT_SECRET, SUPABASE_*, SMTP_*
cp .env.example frontend/.env.local
npm run prisma:migrate              # יוצר מיגרציה למודל User
npm run prisma:seed                 # admin@dershtele.co.il / admin1234
npm run dev:frontend                # :3000
npm run dev:backend                 # :4000
```
